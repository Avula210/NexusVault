How Passkeys Are Stored

✦ Our server stores only: credential ID, your public key (COSE format), and sign count

✦ The sign count increments on every login — a drop would indicate a cloned authenticator

Registration Flow
Step by step
1
Client → Server: Begin Registration
Browser sends username. Server calls generate_registration_options() — creates a random challenge, RP info, and user ID. Stored in server-side session.
2
Browser → Authenticator: Create Key Pair
Browser calls navigator.credentials.create(). Device hardware (TPM / Secure Enclave) generates an ES256 key pair. Private key is stored in hardware — never exported.
3
Client → Server: Finish Registration
Authenticator returns: credential ID, public key (COSE), attestation object, client data JSON. These are sent to the server.
4
Server: Verify & Store
verify_registration_response() checks: challenge matches, origin matches (anti-phishing), RP ID hash matches, user verification flag set. Public key + credential ID stored in Firebase Firestore.
Authentication Flow
Step by step
1
Client → Server: Begin Login
Browser sends username. Server generates a new random challenge and sends it with the list of allowed credential IDs (from DB).
2
Browser → Authenticator: Sign Challenge
Browser calls navigator.credentials.get(). Device performs biometric check, then signs the challenge with the stored private key. Signature never reveals the private key.
3
Client → Server: Submit Signature
Sends: credential ID, authenticator data, client data JSON, signature. Server looks up public key by credential ID.
4
Server: Verify Signature
verify_authentication_response() verifies the signature using the stored public key. Checks sign count > stored (anti-replay). On success, session is created in DB.
Tech Stack
Frontend
  • HTML5 / CSS3 / Vanilla JS
  • WebAuthn API (navigator.credentials)
  • CBOR encoding for binary data
  • Base64url encoding for transport
Backend
  • Python Flask (web framework)
  • py_webauthn 2.x (WebAuthn library)
    • .
  • Flask-Session (server-side sessions)
  • Flask-CORS (cross-origin handling)
  • Firebase Firestore (cloud database)
  • Vercel serverless deployment
Database (Firebase Firestore)
  • users — profile and account metadata
  • credentials — public keys and sign counts
  • sessions — active session validation
  • audit_log — authentication activity
Security Properties
  • No passwords — ever
  • Phishing-proof (origin-bound)
  • Replay-proof (monotonic sign count)
  • Challenge expiry (5 min TTL)
  • Single active session enforcement
  • Device-bound cryptographic credentials
